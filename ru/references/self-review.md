# Финальное само-ревью диффа — три линзы

Читай этот файл после завершения правок кода и ДО отчёта «готово».
Смысл: автор диффа слеп к собственным ошибкам, потому что читает «что хотел
написать», а не «что написано». Три линзы ниже — три разных способа сломать
эту слепоту. Проходи их по очереди, это 5–10 минут.

Перед стартом получи РЕАЛЬНУЮ картину изменений, не по памяти: `git status`
(покажет и untracked-файлы — новые модули `git diff` не видит!) плюс
`git diff HEAD` (покажет staged и unstaged вместе); если по ходу задачи были
промежуточные коммиты — дифф против стартового коммита задачи. Новые untracked
файлы читай целиком как часть «диффа».

## Линза 1. Построчно, с окружением

Для каждого hunk'а:

1. Прочитай **всю функцию/блок**, в котором лежит правка, — не только изменённые
   строки. Ошибки живут на швах: правка верна сама по себе, но конфликтует
   со строкой десятью выше.
2. Для каждой строки, которая обрабатывает вход, условие или границу, спроси:
   что будет при **пустом / null / нуле / отрицательном / очень большом** входе?
   Ноль и пустая строка — falsy: `if (x)` на числе — почти всегда баг.
   (Строки без ветвлений и входов — присваивания, логирование — так не гоняй.)
3. Проверь границы: `<` vs `<=`, off-by-one в индексах и срезах, первая/последняя
   итерация цикла.
4. Проверь пути ошибок: что происходит, когда await падает, когда файл не
   существует, когда ответ не 200? Правка добавила новый путь ошибки — кто его
   обрабатывает?

## Линза 2. Удалённые и изменённые инварианты

Пройди по диффу ещё раз, глядя ТОЛЬКО на удалённые и заменённые строки
(минусы в диффе):

1. Каждая удалённая строка что-то **гарантировала**: проверку, порядок вызовов,
   инициализацию, освобождение ресурса, формат данных.
2. Для каждой спроси: **кто полагался на эту гарантию?** Найди потребителей
   (grep по имени, по полю, по событию), убедись, что они переживут её исчезновение.
3. Изменённые условия (`if` переписан) — проверь оба направления: случаи, которые
   раньше проходили, а теперь нет, и наоборот. Второе направление забывают всегда.

## Линза 3. Трассировка по call sites

Для каждого изменённого публичного контракта (сигнатура функции, формат
возвращаемого значения, имя события, схема данных, ключ конфига):

1. Найди **всех** вызывающих/потребителей: grep по имени во всём проекте, включая
   тесты, другие слои (клиент/сервер, main/renderer/preload) и конфиги.
2. Не забудь **дубликаты кода**: тот же модуль может существовать второй копией
   (форк директории, вендоренный файл, второй бандл). Починить одну копию из
   двух — «исправил, но у пользователя не работает».
3. Динамические вызовы grep не находит: строковые ключи, dependency injection,
   таблицы диспетчеризации, рефлексия. Если проект их использует — проверь руками.

## Вердикт по каждой находке

Не всякая находка требует правки. Классифицируй:

- **Чиню сейчас** — реальный дефект в моём диффе.
- **Сообщаю, не чиню** — дефект вне моего scope (был до меня / соседняя система).
  В отчёт отдельным списком.
- **Ложная тревога** — перечитал, код корректен. Двигаюсь дальше, не «улучшаю».

После фиксов по находкам — прогони проверку (тесты/запуск) ещё раз: правка
по итогам ревью — такая же правка, она тоже может сломать.

## Признак качественного прохода

Если три линзы не дали НИ ОДНОЙ находки и дифф больше 30 строк — проверь себя:
не просмотрел ли ты дифф «по диагонали», подтверждая себя. Пройди Линзу 2 ещё
один раз (удалённые инварианты — самый урожайный угол) — и если снова пусто,
**это валидный результат**: чистый дифф существует. Не выдумывай находку, чтобы
«ревью выглядело проведённым», — ложная находка и правка по ней вреднее пустого
ревью.
