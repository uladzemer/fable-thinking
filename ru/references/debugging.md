# Протокол дебага — полный пересбор после «двух ударов»

Читай этот файл, когда сработало правило «два удара — стоп» (2 неудачные попытки
по одной проблеме) или когда берёшься за баг, который «непонятно откуда».

Смысл протокола: заменить «попробую ещё вот это» на управляемое сужение
пространства гипотез. Каждый шаг либо подтверждает, либо убивает гипотезу —
движение всегда вперёд, даже когда фикс ещё не найден.

## Шаг 0. Заморозь руки

Никаких **фиксов**, пока не пройдены шаги 1–4. Патч, написанный до понимания
причины, — это ставка, а не инженерия. Если ты здесь после «двух ударов» — две
такие ставки ты уже проиграл; если берёшься за незнакомый баг с нуля — не делай
и первой.

Диагностические правки (временные логи, принты, отладочные ассерты) — разрешены
и поощряются: они собирают данные, а не меняют поведение. Помечай их и убирай
перед финальным диффом.

## Шаг 1. Воспроизведение

- Найди **минимальный** способ вызвать симптом. Одна команда лучше десяти кликов.
- Если баг не воспроизводится — это твоя задача №1, не «попробую фикс вслепую».
  Невоспроизводимый баг чинить нельзя: нечем проверить, что починил.
- Зафиксируй точный текст ошибки / неверное значение / скриншот. «Что-то не то» —
  не симптом.

## Шаг 2. Граница «работает / не работает»

Сформулируй пары наблюдений:

- Работает в X, не работает в Y — чем X отличается от Y?
- Работало в коммите A, сломалось к коммиту B — `git log A..B` по затронутым путям.
- Работает с input P, падает с input Q — чем Q особенный?

Граница — самый информативный факт о баге. Часто она одна сразу указывает слой.

## Шаг 3. Таблица гипотез

Выпиши 3–5 гипотез, для каждой — **как её дёшево проверить** и **что бы её опровергло**:

```
H1: стейл-кэш на клиенте        | проверка: hard reload / инкогнито | опровергнет: тот же баг в чистом профиле
H2: гонка init-порядка          | проверка: лог порядка событий     | опровергнет: порядок стабильный и верный
H3: моя правка не доехала       | проверка: grep задеплоенного кода | опровергнет: правка на месте
```

Обязательный пункт в таблице — гипотеза-унижение: «моя правка вообще не
выполняется» (не тот файл, не тот процесс, кэш, не пересобрано, вторая копия кода).
По опыту это причина каждого третьего «непонятного» бага.

## Шаг 4. Эксперименты по одной переменной

Проверяй гипотезы от дешёвой к дорогой, **по одной за раз**. Изменил два фактора,
стало лучше — ты не знаешь, который сработал, и таблица гипотез испорчена.

Инструменты сужения:

- **Печать реальности**: логни фактическое значение/структуру в точке подозрения.
  Спор с реальностью проигрывается всегда — смотри, что там на самом деле.
- **Бинарный поиск по слою**: работает ли на слой ниже? (API отвечает верно →
  баг выше; данные в БД верные → баг в чтении/отображении.)
- **Бинарный поиск по времени**: `git bisect` или временный checkout последнего
  заведомо рабочего коммита. Осторожно: перед любой командой, трогающей working
  tree (`checkout`, `stash`, тем более `reset --hard`), проверь `git status` —
  несохранённые чужие изменения терять нельзя; разрушающие варианты — только
  с явного разрешения пользователя.

## Шаг 5. Причина названа — проверь глубину

Прежде чем писать фикс, ответь: «это причина или ещё один симптом?»
Тест: «если я исправлю здесь, мог бы тот же класс ошибки возникнуть в соседнем
месте?» Если да — причина глубже (валидация на входе, а не проверка на каждом
использовании; контракт модуля, а не каждый вызов).

## Шаг 6. Фикс + регрессия

- Фикс пишется только теперь — и он обычно маленький, потому что бьёт в причину.
- Сначала покажи, что симптом воспроизводится; примени фикс; покажи, что исчез.
  Если есть тесты — добавь тест, который падал бы без фикса.
- Прогони соседние тесты: фикс причины иногда меняет поведение, на которое
  кто-то полагался.

## Анти-паттерны (если делаешь это — вернись к шагу 0)

- Фикс со словами «на всякий случай», «заодно», «должно помочь»
- Два изменения в одном эксперименте
- «Починил», не воспроизведя баг ни разу
- Подавление симптома (try/catch вокруг падения, `|| default` на undefined)
  без ответа, почему значение вообще неверное
- Отладка по памяти о коде вместо чтения текущего кода
